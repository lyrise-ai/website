Welcome to the ROI engine 👋

The ROI report turns a few facts about a prospect's company into a polished, McKinsey-styled financial report — researched, modelled, written and delivered as a PDF. It is the most intricate subsystem in the website repo, and it got here through three distinct lives. Knowing that history is the fastest way to understand why the code looks the way it does.

🧭

The one-sentence mental model

A user describes their company → an AI agent researches it, builds a deterministic financial model, and writes the report copy → the copy is poured into an HTML template → the template is rendered to a PDF and emailed. Everything else is detail.

The journey

Three lives of the ROI report

Same goal every time — a credible ROI report — but the machine behind it was rebuilt twice. Each era solved a real problem and left fingerprints in today's code.

1

The n8n workflow — ROIGEN (1).json

Era one · the assembly line

The first version lived entirely in n8n: a visual, node-by-node pipeline triggered by a webhook. Each step was its own box on a canvas — three chained GPT-4o-mini calls (each locked to a strict JSON schema), with deterministic JavaScript "Code" nodes doing the maths and glue between them.

Webhook → Normalize Input → Company Analyst → Parse → ROI Modeler → Parse → ROI Calculator → Report Writer → Parse → Assemble Report → HTML Template → Email PDF

It worked, but it was fire-and-forget: no web research, no way to edit a report after generation, and the logic was trapped inside a JSON export that no one could code-review or test.

Inheritance → the staged shape Inheritance → deterministic ROI Calculator Inheritance → JSON-schema'd LLM calls
2

The Claude template — template_instructions.txt

Era two · the craftsmanship

In parallel, the team ran reports by hand: paste a meeting transcript into Claude alongside a long master prompt — the "ROI Report Master Instructions v3.2" — and Claude would produce a finished .docx. This era was about quality, not automation. The template still defines our standards:

  • A silent 6-phase reasoning workflow (intelligence vectoring → financial modelling).
  • Financial rules 6A–6E: sourced blended rates, the 5–20% revenue sanity check, full arithmetic chains, internal consistency cross-checks.
  • A locked 17-section report structure and a McKinsey-grade design system (navy cover card, one accent blue, no vertical table borders).
  • The vocabulary we still use: "Operational Dividend", "Total Financial Gain", "Hours Returned" — never "cost savings" or "ROI".

Powerful, but fully manual — one skilled person, one prompt, one report at a time.

Inheritance → the financial rulebook Inheritance → section structure & design tokens Inheritance → tone & terminology
3

The in-repo agent — src/lib/roi/ · today

Era three · the product (you are here)

The current system fuses both: n8n's reliable staged pipeline and the Claude template's rigour — now as real, testable TypeScript inside this Next.js app. The chain of n8n nodes collapsed into a single unified streamText agent (src/lib/roi/agent.ts) that researches, models and writes in one loop, calling tools instead of hopping between canvas boxes.

POST /api/roi-agent → agent loop { web_search · fetch_page · run_financial_model · set_report_copy } → assembleReport → renderTemplate → SSE stream + Supabase + PDF

It gained everything the earlier eras lacked: live web research, conversational editing, streaming progress, persistence, and an automated quality eval harness.

Live endpoint · streams SSE Generate + chat-edit Researches the web
How it works today

Anatomy of the current system

The whole engine lives under src/lib/roi/ (TypeScript) and is reached through one route. Here is the map you'll navigate daily.

WhereWhat it does
pages/api/roi-agent.jsThe live endpoint. Handles mode:'generate' and mode:'chat', maps the form payload, streams SSE, persists to Supabase. 5-minute serverless timeout.
src/lib/roi/agent.tsThe brain. One streamText loop with ~13 tools that mutate a single ReportState object in place.
pipeline/normalize.tsAccepts both structured processes[] and legacy flat form fields — the n8n "Normalize Input" node, reborn.
pipeline/roiCalculator.tsPure, deterministic financial maths. No LLM. This is the n8n "ROI Calculator" node's direct descendant and enforces template rules 6B–6E.
pipeline/assembleReport.tsTurns state into every {{$json.display.*}} variable — the n8n-style placeholders survived all three eras.
pipeline/renderTemplate.tsPours those variables into public/roi-template.html.

tools/webSearch (Tavily → Brave → Jina fallback) and fetchPage (Puppeteer).

⚠️

Two gotchas worth knowing on day one

1. src/lib/roi/ is TypeScript; the rest of the repo is JS. The @components-style webpack aliases don't work in here — use @/src/... imports.   2. If you change an SSE payload in the agent, update its consumer src/lib/drainSSE.js too, or the UI goes quiet.

The toolbox

Main features

What the engine can do — each of these is a feature you may be asked to extend.

🪄

One-shot generation

mode:'generate' — describe a company, get a full researched report. The agent runs research → model → copy in a single loop.

💬

Conversational editing

mode:'chat' — refine a finished report by chatting. Tools like update_workflow and scale_rates mutate it live.

🔎

Live web research

The agent searches the web for company facts and real salary benchmarks (Glassdoor, Bayt, Gulf Talent…) instead of guessing rates.

🧮

Deterministic model

roiCalculator does all maths in plain TS — same inputs, same numbers, every time. The financial rulebook is enforced here.

📡

SSE streaming

Progress, partial reports and chat replies stream to the browser in real time via server-sent events.

💾

Persistence & sharing

Every report is saved to Supabase and reopened at /report/[id] — viewable and editable later.

📄

PDF + email delivery

/api/roi-pdf renders the HTML to PDF; /api/roi-email sends it through Resend.

📊

Bulk mode + evals

roi/bulk/ turns an Apollo CSV into many reports; evals/roi/ scores report quality as a regression test.

🔐

Access rules to keep in mind

Non-employees can generate 1 report and send 5 chat messages. Employees (role EMPLOYEE or an @lyrise.ai email) are unlimited. ROI pages assume an authenticated Supabase user.

Your turn

How to add a new feature

Most new work falls into one of four shapes. Find yours, follow the thread end to end, and add an eval case so it stays working.

A

A new agent capability → add a tool

Add a tool({ ... }) in agent.ts. Copy an existing one (update_workflow is a good template): give it a clear description, a zod inputSchema, and an execute that mutates ReportState in place — never return a new state object. Re-assembly and the SSE update happen for you.

B

A new report section → template + assemble

Add the markup and {{$json.display.yourKey}} placeholders to public/roi-template.html, then produce yourKey in assembleReport.ts. If the agent needs to write that copy, add a field to ReportCopy in types.ts and to the set_report_copy / update_copy tools.

C

A new model input → follow the data spine

A new per-workflow number flows through types.ts → roiCalculator.ts (use it in the maths) → assembleReport.ts (display it). A new form field flows through pages/roi-report.jsx → mapFormToPayload() in the API route → normalize.ts.

D

New behaviour / better output → tune the prompts

Reasoning, tone and rules live in src/lib/roi/prompts/. The template_instructions.txt master spec is the source of truth for what a good report contains — keep prompt edits aligned with it.

✅

Before you open a PR

Run npm run lint (the build won't catch lint errors — a Husky pre-commit hook does). Test fast with devOptions.skipLLM or estimatesOnly to skip slow LLM calls. Add or update a case in evals/roi/cases/ and run npm run eval:roi. Commit only redacted or synthetic client data.
